# AGENTS.md

## Repo purpose

This repository is `TimelineForAudio`.
Its job is to turn local audio files into timeline-oriented artifacts for human review and LLM handoff.

## Must preserve

- Keep the product local-first.
- Treat CLI / worker / output artifacts as the source of truth. There is no web UI in the current product scope.
- Do not delete, overwrite, mass-move, or mass-rename original input audio files.
- Keep gaps, warnings, partial failure, and fidelity loss visible instead of hiding them.
- Preserve the current run/output contract unless a breaking change is explicitly approved.
- Prefer fixed input/output/state roots by default. Configurable paths are fine, but arbitrary ad-hoc path handling should not become the normal workflow.

## Product-specific guardrails

- `IPA` is a core intermediate. Do not casually remove or bypass that role.
- Preserve `request.json`, `status.json`, `result.json`, `manifest.json`, and export package consistency.
- If file organization or rename behavior is added later, record old path, new path, hash, mtime, and size in a manifest-like artifact.
- Keep Docker / `docker compose` / `start.bat` reproducible.

## Standard model

```text
InputSource = configured input source
InputItem   = detected processable input from InputSource
Job         = processing request based on an InputItem snapshot and profile
Run         = one execution attempt for a Job
Artifact    = output generated by a Run
```

## Safe work without extra confirmation

- read-only investigation
- `AGENTS.md`, README, docs, and `.env.example` updates
- non-destructive small code fixes
- lightweight local tests or smoke checks
- Docker build / compose smoke checks
- output contract consistency fixes that do not remove user data

## Ask before doing these

- deleting or rewriting raw input
- mass move / mass rename of user files
- breaking output contract changes
- repo or product rename
- new hosted/cloud dependency
- deploy, external posting, or secret changes
- broad architecture rewrites

## Before finishing

- Read the README and relevant docs first.
- Keep tests and current behavior stable unless the task explicitly changes them.
- Update docs when behavior or contracts change.
